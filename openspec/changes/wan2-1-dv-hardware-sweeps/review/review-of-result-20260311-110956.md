## Overall Assessment

The change largely delivers what it promises: the sweep is device-selectable, artifacts are device-scoped, metadata includes normalized device assumptions, DP semantics are preserved, and consolidated DV stakeholder reporting is generated from machine-readable results with compatibility checks.

The main issues I see are not “math is wrong” but “semantics and presentation can mislead”: the “1-second SLA” concept is used in a way that mixes latency framing with throughput-oriented sizing ratios, and the at-a-glance DV table emphasizes MemIO gap even for the compute-limited tier (DV300), which can be misread. There is also a small but real markdown formatting bug in the generated DV stakeholder report (missing newline before the first image in each device section).

## Correctness Review (Implementation)

- Device selection and output layout look correct and match the spec: `device_run_dir(.../reports/hardware_sweeps/<device>/<run_id>/)` and `comparison_run_dir(.../reports/hardware_sweeps/comparisons/<comparison_run_id>/)` are used consistently.
- The “model-local boundary” requirement is met in spirit: the device resolver is implemented in `extern/modelmeter/models/wan2_1/hardware_sweep.py` while reusing shared device definitions (`modelmeter.devices.gpu.DV*`, `NGU800P`) without editing them.
- DP semantics are preserved and explicitly recorded as `effective_gpus=min(batch_size, device_num)` in the runner (`_dp_cost_seconds`) and propagated into CSV/JSON; I sanity-checked `dv200/20260311-dv200/results.csv` and found 0 mismatches across 2048 rows.
- Metadata quality is strong for reproducibility: `results.json` includes precision settings, util profiles, the workload grid, batch/device grids, run id, script path, platform/python, and a git revision (`git_rev_modelmeter`) for the modelmeter subtree.
- Backward compatibility is handled cleanly with thin wrappers: `run_ngu800p_concurrency_sweep.py` and `run_make_ngu800p_sweep_figures.py` simply call the generic entrypoints with `default_device="ngu800p"`.

## Correctness Review (Produced Results)

- Artifact completeness is good for all four device runs (`ngu800p`, `dv100`, `dv200`, `dv300`): each run has `results.json`, `results.csv`, `summary.md`, and the three standard figures under `figures/`.
- The consolidated comparison bundle is complete: `comparison-table.csv`, `stakeholder-report.en.md`, per-device prefixed figures (`dv100_*`, `dv200_*`, `dv300_*`), cross-device comparison figures (`*_compare.svg`), and the shared DiT diagram copy.
- Key numeric sanity checks pass internal consistency: the batch=1 latencies line up exactly with the per-video “required” rates vs device peaks (DV100 and DV200 MemIO-limited; DV300 tensor-limited), and the NGU baseline run’s batch=1 latency for the most demanding input (458.872s at 1280*720/81/50) matches the legacy NGU stakeholder report’s appendix numbers.

## Stakeholder Report Review (Soundness and Clarity)

Strengths:

- Structure is aligned with the NGU report and meets the spec: header, executive summary, shared methodology/diagrams, per-device analysis sections, appendix tables, and cross-device conclusions/figures.
- The report keeps the key caveat that this is a first-order analytic sizing model and should be validated with profiling.
- Compatibility checks exist and are meaningful: precision and workload grid metadata are compared before combining DV runs, and mismatch errors are reported with specific dimensions.

Issues and risks:

- “1-second SLA” semantics are mixed in a way that can confuse readers: the report uses batch=1 latency to talk about “closest to 1-second target”, but the “Required MemIO for 1s” and “Gap vs 8-GPU peak” numbers are effectively throughput-oriented capacity ratios against an 8-device node peak under the DP-across-requests model (no intra-request sharding), not a direct statement that “8 GPUs gets you closer to 1s latency”.

> DECISION: This is a real problem, but it is a report-semantics problem rather than a modeling-correctness problem.
>
> FIX: Keep `Latency at batch=1 (s)` as the latency metric, rename the current `1s SLA` figure/title/column language to something like `1s normalized per-video resource requirement` or `1s capacity-gap estimate`, and add one explicit note that comparing required resource rate to an 8-GPU peak is a capacity-gap heuristic under DP-across-requests rather than a claim that 8 GPUs make per-video latency approach 1 second.
>
> RATIONALE: The math is internally consistent and matches the legacy NGU report framing, but the current wording mixes single-request latency with normalized resource-rate sizing and becomes easier to misread once devices are compared side by side.

- The at-a-glance comparison table reports only MemIO requirement/gap even for DV300 where the primary bottleneck is tensor compute; the “Gap vs 8-GPU peak” column can be misread as “how close to the target this tier is”, but for DV300 the compute gap (4.83x vs 8-GPU peak) is more constraining than the MemIO gap (2.12x).

> DECISION: This is a real problem for DV300 and should be corrected.
>
> FIX: Change the overview table and generated summary so it either shows both `required_tensor_vs_peak_8gpu` and `required_memio_vs_peak_8gpu`, or replaces the single gap column with a bottleneck-aligned `Required/peak gap` that explicitly uses the dominant resource for each device and names that resource in the table cell or adjacent column.
>
> RATIONALE: `comparison-table.csv` already shows that DV300 is tensor-limited (`4.83x` tensor gap vs `2.12x` MemIO gap), so the current table overemphasizes a secondary constraint and can misstate which hardware delta is actually decisive.

- There is a markdown formatting bug in the generated DV stakeholder report: the first throughput figure in each device section is glued to the preceding sentence with no newline (lines 128/191/254 contain `...MemIO).![](figures/...)`), which affects readability and can render oddly in some viewers.

> DECISION: This is a real bug.
>
> FIX: Make the report writer guarantee a fresh block boundary before inserting images, ideally in `_add_image` or the surrounding paragraph/image emission path, then regenerate the stakeholder report artifact.
>
> RATIONALE: The generated Markdown literally contains `...MemIO).![](figures/...)` or `...compute.![](figures/...)`, which is an output-format defect independent of any modeling assumptions.

- The report references an external mirror URL in “Source:” text; that is fine, but if the repo intends reports to be fully offline-viewable, consider whether this should be moved to a footnote or an internal reference.

> DECISION: This is not a meaningful problem for this change and does not block accepting the result.
>
> FIX: Treat this as optional cleanup only: move the URL to a footnote if desired, or add a short note that the image itself is copied into the generated bundle for offline reading.
>
> RATIONALE: The report already vendors `wan2_1_video_dit_arch.jpg` into the comparison output, so the external link is attribution context rather than a runtime dependency for viewing the report.

## Recommendations (Non-Blocking but High Leverage)

- Clarify the meaning of “1s SLA” everywhere (report text, figure titles, and column labels): decide whether it is a latency SLA (per-video latency at batch=1) or a throughput/capacity target (1 video/s at saturation), or explicitly present both as distinct metrics.

> DECISION: Accept.
>
> FIX: Apply this consistently across the executive summary, the at-a-glance table, the per-device conclusions, and the figure titles so the latency/capacity distinction is explicit in every place a stakeholder might skim.
>
> RATIONALE: Fixing only one section would leave the same ambiguity elsewhere because the current wording is repeated across multiple report surfaces.

- Adjust the at-a-glance DV table to avoid MemIO-only bias: include both compute and MemIO gaps, or compute a single “required/peak” gap aligned with the primary bottleneck for each tier.

> DECISION: Accept.
>
> FIX: Prefer a bottleneck-aligned `Required/peak gap` plus a separate `Primary bottleneck` column if we want the stakeholder table to stay compact; otherwise include both compute and MemIO gaps in full.
>
> RATIONALE: The current compact table is useful, but it should not achieve brevity by hiding the decisive constraint on DV300.

- Fix the markdown newline issue in `run_make_dv_stakeholder_report.py` (the helper that appends raw blocks/images should ensure it starts on a fresh line).

> DECISION: Accept.
>
> FIX: Repair the block/image emission path and regenerate the produced stakeholder report so the checked-in artifact no longer contains the formatting defect.
>
> RATIONALE: This is a straightforward output bug with a clear local fix and no design tradeoff.

- Consider adding a small unit test surface for the new helpers: `resolve_device_profile` (metadata derivations, selector validation), and `compute_device_summary` (bottleneck classification and saturation point invariants) so regressions in reporting semantics are caught without needing to regenerate full reports.

> DECISION: Worth doing, but not required to accept the current result review.
>
> FIX: Add focused tests for selector validation, derived metadata fields, bottleneck classification, and saturation-point invariants in a follow-up change or when the report semantics are updated.
>
> RATIONALE: The current review already established that the produced artifacts are internally consistent; tests would reduce regression risk, but their absence does not by itself invalidate the current implementation or outputs.

## Open Questions / Assumptions to Confirm

- Are the DV device peaks (DV100/DV200/DV300) considered authoritative for stakeholder decision-making, or are they placeholders for modeling? If placeholders, the stakeholder report should explicitly say so in the header section to avoid over-trust.

> DECISION: Treat them as preliminary modeling inputs rather than authoritative stakeholder-ready hardware specifications.
>
> FIX: Add a header note stating that DV100/DV200/DV300 numbers come from the current shared device registry and are used as first-pass analytic assumptions pending independent validation or vendor-confirmed specs.
>
> RATIONALE: The proposal already describes the DV aliases as preliminary, and `extern/modelmeter/devices/gpu.py` explicitly marks `cuda_tflops` as a dummy compatibility field and labels DV300 as TBD-like, so the report should not imply stronger certainty than the inputs warrant.

- The resolver derives bisection bandwidth as `4x p2p` when not provided by the shared device spec; this is currently low-impact because model communication is effectively zero in these sweeps, but if future model modes introduce comm costs, this assumption may become significant and should be revisited or made configurable.

> DECISION: This is not a current problem for these published results.
>
> FIX: Document the assumption in metadata or report notes and defer configurability until a future Wan2.1 mode introduces non-trivial communication cost into the bottleneck calculation.
>
> RATIONALE: The current sweeps are effectively communication-free, so this derived field does not materially affect the reported bottlenecks, throughput curves, or stakeholder conclusions in this artifact set.
